**For** 运动损伤和慢性病患者  
**Who** 需要居家康复指导但难以获得持续的专业支持

## 2. 项目背景与问题陈述

### 2.1 行业现状与痛点

运动损伤康复和慢性病管理是现代社会的重大健康挑战。随着久坐办公人群增加，颈椎病、腰椎间盘突出等疾病发病率持续上升。传统康复治疗存在以下核心痛点：

| 问题维度 | 具体表现 |
|---------|--------|
| **专业资源稀缺** | 康复医师数量严重不足，患者每周仅能获得 1-2 次面对面指导 |
| **居家康复缺乏依据** | 患者出院后缺乏科学指导，运动不足或运动过量均可能导致二次损伤 |
| **康复方案同质化** | 传统康复计划多为通用模板，难以针对个体差异进行个性化调整 |
| **依从性监测困难** | 居家训练质量无法保证，动作不规范难以及时发现 |
| **康复周期漫长** | 慢性病康复往往需要数月持续干预，患者容易中途放弃 |

### 2.2 政策与市场机遇

- **政策利好**：《"健康中国 2030"规划纲要》明确提出发展康复医学
- **市场规模**：2025 年中国康复医疗器械市场规模超过 1500 亿元
- **技术成熟**：姿态估计算法已达医疗级水准，多模态大模型实现深度融合
- **数据基础**：Rehab-Pile 等标准化康复动作数据集为 AI 康复处方提供训练基础

### 2.3 项目价值

本项目致力于打通 **"用户评估 → 智能处方生成 → 动作示范 → 实时纠正 → 效果追踪"** 的完整康复闭环，让运动损伤和慢性病患者在家也能获得专业级的康复指导，实现"将康复治疗师装进手机"的愿景。

### 6.2 核心技术栈

| 层级 | 技术选型 | 说明 |
|------|---------|------|
| **前端** | 原生 HTML5 + CSS3 + JavaScript | 无框架依赖的静态 SPA，零构建部署 |
| **前端姿态** | MediaPipe Pose（33 关键点） | 模拟关键点演示模式，待接入真实流 |
| **后端框架** | FastAPI（Python 3.13+） | 原生异步，自动 Swagger 文档，性能优异 |
| **鉴权** | 自定义 JWT（PBKDF2-HMAC-SHA256） | 支持 user/doctor/admin 三级角色 |
| **关系数据库** | 开发：SQLite / 生产：PostgreSQL 17 | 通过 `DATABASE_URL` 一键切换 |
| **ORM** | SQLAlchemy（`Base.metadata` 启动建表） | 12 张业务表统一建模 |
| **缓存** | Redis 8（可选，`REDIS_URL` 未配时自动禁用） | 热点数据 JSON 缓存 |
| **对象存储** | 开发：本地 / 生产：MinIO | 双后端抽象，影像报告存储 |
| **向量检索** | Chroma / Qdrant / local JSON | 三后端 RAG 知识检索 |
| **大模型** | DeepSeek（火山方舟 OpenAI 兼容接口） | 处方生成 + 本地安全兜底 |
| **姿态推理** | RTMPose ONNX（`models/`） | 规则算法 + 深度学习双模式 |
| **3D 可视化** | 关键点矩阵运算生成骨骼数据 | 骨骼规范 + AR 叠加信息 |

### 6.3 核心调用链路

**处方生成链路：**
```
前端问诊表单 → FastAPI /api/generate_prescription
  → 读取患者档案 + 影像风险等级
  → validators 检测红旗症状（高风险则拦截）
  → RAG 检索知识上下文（actions.json + 向量索引）
  → safety 模块过滤/降阶高风险动作
  → DeepSeek 大模型/本地兜底生成处方摘要
  → 写入 prescriptions + actions 表
  → 返回处方摘要与动作列表
```

**姿态纠正链路：**
```
前端摄像头/模拟关键点 → /api/correct_pose
  → algorithms.py 计算角度和距离
  → 返回 feedback / score / status
  → spatial.py 生成 3D 骨骼 + AR 叠加
  → voice_feedback.py 生成语音提示文本
```

**训练打卡与报告链路：**
```
患者训练打卡 → /api/training_checkins
  → 写入 training_checkins 表
  → progress_reports.py 生成周报/月报/custom
  → /trends 日期聚合 /visualization 统计可视化
```

## 9. 风险评估与应对策略

### 9.1 风险矩阵

| 风险编号 | 风险描述 | 概率 | 影响 | 等级 | 应对策略 |
|---------|---------|:---:|:---:|:---:|---------|
| **R1** | 大模型接口依赖外部服务，可能出现网络延迟或鉴权失败 | 中 | 高 | 🔴 高 | **降低**：实现本地兜底摘要机制，API 失败时自动降级，保证演示链路不中断 |
| **R2** | 动作纠正算法覆盖动作有限（当前仅 2 个），无法覆盖知识库所有动作 | 中 | 中 | 🟡 中 | **缓解**：优先实现核心高频动作算法，演示时说明扩展计划 |
| **R3** | 前端真实视频关键点提取尚未接入 MediaPipe，目前依赖模拟关键点 | 中 | 中 | 🟡 中 | **接受**：模拟关键点可验证后端接口完整性，真实提取作为后续迭代 |
| **R4** | 康复处方直接涉及用户健康，大模型输出可能存在医学风险 | 高 | 高 | 🔴 高 | **规避**：多层安全校验（红旗症状拦截、动作安全过滤、禁忌症映射、免责声明、提示词约束） |
| **R5** | SQLite 并发性能有限，不适合生产环境多用户访问 | 低 | 中 | 🟡 中 | **转移**：通过 `DATABASE_URL` 一键切换 PostgreSQL 17，已含 docker-compose.yml |
| **R6** | GitHub 访问依赖网络和代理环境 | 中 | 低 | 🟢 低 | **接受**：提供 Gitee 备选方案，本地保留完整代码备份 |
| **R7** | 团队人员变动或协作冲突 | 低 | 高 | 🟡 中 | **缓解**：代码模块解耦，每人有独立工作路径，Git 记录可追溯 |
| **R8** | 医疗数据隐私合规风险 | 低 | 高 | 🟡 中 | **规避**：本地优先处理视频帧，不上传原始视频仅传关键点坐标 |

### 9.2 已实施的安全措施

- ✅ **红旗症状检测**（`validators.py`）：检测 10+ 高风险关键词
- ✅ **影像风险拦截**：`high` 风险阻止处方生成，强制就医
- ✅ **动作安全过滤**（`safety.py`）：根据禁忌症自动降阶高风险动作
- ✅ **处方免责声明**：所有处方标注"仅供参考，请遵医嘱"
- ✅ **密码 PBKDF2-HMAC-SHA256 加盐哈希**，禁止明文存储
- ✅ **JWT Secret 环境变量配置**，不提交代码仓库
- ✅ **权限隔离**：用户隔离、医患关系绑定、角色分级
- ✅ **大模型输出约束**：提示词要求不编造动作、安全优先
