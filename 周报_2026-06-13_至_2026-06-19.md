## 一、本周工作概述

本周主要围绕系统核心功能完善、前后端联调、用户数据隔离、动作纠正算法优化和提示词模板调整展开。项目已完成可运行的 MVP，并在此基础上继续补充登录认证、用户处方绑定、处方展示优化、姿态反馈优化和大模型提示词规则约束。

当前系统可以支持用户填写基本信息、主诉、疼痛部位和活动度评分，后端根据结构化康复动作知识库检索候选动作，并调用豆包大模型生成个性化康复处方。系统同时提供姿态纠正接口，用于根据关键点数据返回动作反馈、评分和状态。

## 二、本周完成内容

### 1. 用户认证与数据绑定

本周完成 JWT 登录相关功能集成，支持用户注册、登录和身份认证。后端将处方记录与当前登录用户进行绑定，解决了历史处方查询时不同用户数据混杂的问题，为后续患者档案、训练记录和个性化历史追踪打下基础。

### 2. 后端接口与前后端联调

继续完善 FastAPI 后端接口，并配合前端完成真实接口联调。重点验证了处方生成、历史处方查询、动作反馈和页面状态恢复等流程。针对前端请求后端时出现的跨域问题，调整 CORS 配置，保证本地演示环境下前端可以正常访问后端 API。

### 3. 豆包大模型与提示词模板优化

维护豆包/火山方舟大模型调用链路，保证处方生成接口在模型正常返回时可以解析 JSON 结果，在模型异常或网络异常时可以使用本地兜底摘要，避免演示流程中断。

根据评审反馈，本周更新了提示词模板规则：当主诉、疼痛部位和活动度评分存在冲突时，以用户勾选的疼痛部位作为动作选择依据；主诉用于描述症状细节；活动度评分只影响强度表述。这一调整提高了大模型处方生成的稳定性和字段解释一致性。

### 4. 姿态纠正算法优化

配合算法模块优化多个较复杂动作的纠正算法，增强反馈信息和评分逻辑。系统能够根据动作 ID、姿态关键点和可见度信息返回反馈建议、分数和状态，为训练跟练过程提供基础的动作质量判断能力。

### 5. 前端体验与演示流程支持

配合前端完成页面刷新状态恢复、处方 recap、移动端图片显示和训练流程衔接等优化。优化后，用户从问诊输入、生成处方、查看动作卡片到进入训练反馈的流程更加连贯，适合用于阶段演示。

### 6. GitHub 协作与文档整理

处理 GitHub 分支同步、提交和推送问题，确保本地代码与远程仓库保持一致。本周同步整理工作日志和周报，为阶段性检查、答辩材料和项目过程文档提交做准备。

## 四、遇到的问题与解决办法

### 问题 1：本地分支与 GitHub 远程分支存在差异

多人协作过程中，远程仓库存在其他成员提交，本地分支也有新增提交，直接推送可能失败。

解决办法：先检查 `git status` 和提交记录，再通过拉取远程更新、合并或 rebase 的方式同步分支，最后再推送到 GitHub。

### 问题 2：前端调用后端接口存在跨域问题

在前端真实后端模式下，请求 FastAPI 接口时受到 CORS 限制，影响处方生成流程联调。

解决办法：调整后端 CORS 配置，并通过本地前端页面反复验证接口请求是否正常。

### 问题 3：大模型对用户字段的理解可能不稳定

当主诉文字、疼痛部位勾选和活动度评分之间存在冲突时，模型可能把主诉中的描述作为动作选择主要依据，导致推荐动作不符合用户勾选部位。

解决办法：在提示词模板中明确规则：疼痛部位冲突时以用户勾选的疼痛部位为动作选择依据，主诉只描述症状细节，活动度仅影响强度表述。

### 问题 4：GitHub 网络连接不稳定

本地执行 `git pull` 或 `git push` 时出现 GitHub 443 端口连接失败，影响代码同步。

解决办法：检查代理软件和 Git 代理配置，确认代理端口可用后再执行拉取和推送。
