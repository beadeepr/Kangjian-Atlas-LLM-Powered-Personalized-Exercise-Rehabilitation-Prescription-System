# 康健图谱需求分析：数据库设计与功能详细说明

> 版本：v2.0 | 更新日期：2026-06-24  
> 本文档基于当前项目真实代码（`backend/app/models.py`、`backend/app/api.py` 等）整理，采用"需求规划 + 已实现情况"并行写法，全面描述系统数据库设计、功能模块、核心数据流与非功能需求。

---

## 1. 项目概述

**系统名称**：康健图谱 — 基于大模型的个性化运动康复处方生成系统

**系统定位**：面向患者、医生和管理员三类角色，通过前端采集患者主诉、疼痛部位、活动能力、训练记录、影像报告和穿戴设备指标，后端结合结构化康复动作库、RAG 知识检索、DeepSeek 大模型、姿态识别与规则算法，为患者生成康复处方、训练反馈和阶段性康复报告。

**技术架构**：

| 层级 | 技术选型 |
|---|---|
| 前端 | 静态 HTML + CSS + JavaScript，MediaPipe Pose 关键点检测 |
| 后端 | FastAPI（Python），JWT 鉴权，Pydantic 数据校验 |
| 关系数据库 | 开发：SQLite；生产：PostgreSQL 17 |
| 缓存 | Redis（可选，`REDIS_URL` 未配置时自动禁用） |
| 对象存储 | 开发：本地 `backend/uploads`；生产：MinIO |
| 向量检索 | Chroma / Qdrant / local JSON fallback（RAG 知识检索） |
| 大模型 | DeepSeek / 火山方舟豆包（OpenAI 兼容接口），本地兜底处方 |
| 姿态识别 | RTMPose ONNX 模型、MediaPipe Pose 关键点、规则算法 |
| ORM | SQLAlchemy，模型集中于 `backend/app/models.py` |

## 2. 数据库需求与设计

### 2.1 数据库建设目标

系统数据库需要满足以下核心需求：

1. **用户体系**：保存患者、医生、管理员账号、角色、密码哈希和基础资料。
2. **患者档案**：保存患者姓名、年龄、性别、身高体重、疼痛部位、既往史、过敏史、手术史和康复目标。
3. **处方生成**：保存用户主诉、病史、生成摘要、动作明细、大模型原始响应、安全评估和 RAG 上下文。
4. **训练闭环**：保存训练打卡、训练动作、疼痛前后评分、难度评分、完成组数和趋势统计。
5. **影像与风险识别**：保存影像报告文件路径、OCR 文本、红旗症状、风险等级和备注。
6. **穿戴疲劳监测**：保存心率、静息心率、HRV、血氧、运动时长、主观疲劳和系统生成的疲劳风险。
7. **医患协同**：保存医生与患者关联、处方审核、医生调整建议、系统自动调整建议和患者决策结果。
8. **后台管理**：保存用户反馈、管理处理状态，并为管理端统计提供数据来源。
9. **部署扩展**：开发阶段快速使用 SQLite，生产阶段切换 PostgreSQL，配合 Redis、MinIO、向量存储组成完整生产存储栈。

### 2.2 数据库选型

| 场景 | 规划需求 | 当前实现 |
|---|---|---|
| 本地开发 | 快速启动、低配置、无需额外数据库服务 | SQLite，默认路径 `backend/database.db` |
| 生产部署 | 多用户并发、稳定事务、可持久化 | PostgreSQL 17，`docker-compose.yml` 已配置 |
| ORM 管理 | 统一模型定义，减少手写 SQL | SQLAlchemy ORM，模型位于 `backend/app/models.py` |
| 数据初始化 | 应用启动时自动建表 | `database.py` 中 `init_db()` 调用 `Base.metadata.create_all()` |
| SQLite 兼容 | 兼容旧版本本地数据库 | `_ensure_sqlite_columns()` 为旧表补充新增字段 |
| 缓存 | 热点数据缓存 + 健康检查 | Redis（`REDIS_URL` 未配置时自动禁用） |
| 对象文件 | 影像报告、图片、PDF 等上传文件 | 开发用本地 `backend/uploads`，生产用 MinIO |
| RAG 向量数据 | 康复知识检索索引 | Chroma、Qdrant、local JSON fallback |

### 2.3 生产存储栈说明

#### 2.3.1 开发环境 SQLite

```
DATABASE_URL=sqlite:///backend/database.db
```

- 使用 SQLAlchemy `create_engine`，`check_same_thread=False`
- 会话管理：`SessionLocal = sessionmaker(...)`
- 启动初始化：FastAPI `startup` 阶段调用 `init_db()`
- SQLite 补列：`_ensure_sqlite_columns()` 保证旧表与新模型字段兼容

#### 2.3.2 生产环境 PostgreSQL

```
DATABASE_URL=postgresql+psycopg://kangjian:kangjian_password@postgres:5432/kangjian_atlas
```

- 承载所有核心业务数据
- 适合多用户并发、事务一致性、索引查询和后续数据分析
- `docker-compose.yml` 中已配置 `postgres:17` 服务

#### 2.3.3 Redis 缓存

由 `backend/app/cache.py` 封装：

- 为热点数据提供 JSON 缓存
- 通过 `/ready` 和 `/api/deployment/info` 提供健康检查状态
- `REDIS_URL` 未配置时自动返回 disabled，不影响本地开发

```
REDIS_URL=redis://redis:6379/0
```

#### 2.3.4 MinIO 对象存储

由 `backend/app/object_storage.py` 封装：

| 模式 | 环境变量 | 用途 |
|---|---|---|
| 本地文件 | `OBJECT_STORAGE_BACKEND=local` | 开发阶段保存到 `backend/uploads` |
| MinIO | `OBJECT_STORAGE_BACKEND=minio` | 生产阶段保存到 `kangjian-atlas` bucket |

文件路径写入 `imaging_reports.file_path`，数据库只保存元数据和访问引用。

#### 2.3.5 RAG 向量存储

由 `backend/app/rag.py` 管理，支持三类向量存储：

| Provider | 配置 | 用途 |
|---|---|---|
| Chroma | `RAG_VECTOR_PROVIDER=chroma`，`CHROMA_PERSIST_DIR=knowledge/rag_store/chroma` | 本地持久化，适合单机部署 |
| Qdrant | `RAG_VECTOR_PROVIDER=qdrant`，`QDRANT_URL` 或 `QDRANT_PATH` | 服务化向量库，适合生产扩展 |
| local | `RAG_VECTOR_PROVIDER=local` | 本地 JSON 索引，便于测试和演示 |
| auto | `RAG_VECTOR_PROVIDER=auto` | 优先选 Chroma/Qdrant，否则回退 local |

## 3. 数据库表结构设计

当前后端实际定义 **11 张业务表**，均位于 `backend/app/models.py`。

### 3.1 `users`（用户表）

**用途**：保存系统登录账号、角色和基础身份信息，支持患者、医生和管理员。

| 字段 | 类型 | 约束 / 索引 | 说明 |
|---|---|---|---|
| id | Integer | 主键，索引 | 用户 ID |
| account | String(64) | 唯一，非空，索引 | 登录账号 |
| password_hash | String(256) | 非空 | PBKDF2-HMAC-SHA256 密码哈希 |
| nickname | String(64) | 非空 | 昵称 |
| role | String(32) | 非空，默认 `user` | 角色：`user`、`doctor`、`admin` |
| gender | String(16) | 可空 | 性别 |
| age | Integer | 可空 | 年龄 |
| created_at | DateTime | 默认当前时间 | 创建时间 |
| updated_at | DateTime | 自动更新 | 更新时间 |

**关联关系**：

- `users` → `patient_profiles`：一个用户可维护多个患者档案（1:N）
- `users` → `prescriptions`：一个用户可拥有多张处方（1:N）
- `users` → `training_checkins`：一个用户有多条打卡记录（1:N）
- `users` → `imaging_reports`：一个用户可上传多个影像报告（1:N）
- `users` → `wearable_metrics`：一个用户有多条穿戴指标（1:N）
- `users` → `doctor_patient_links`：医生角色与患者关联（1:N，两向）
- `users` → `user_feedback`：一个用户可提交多条反馈（1:N）

---

### 3.2 `patient_profiles`（患者档案表）

**用途**：保存患者康复档案，同一账号可维护多个患者资料（如家庭成员代管）。

| 字段 | 类型 | 约束 / 索引 | 说明 |
|---|---|---|---|
| id | Integer | 主键，索引 | 档案 ID |
| user_id | Integer | 外键 `users.id`，非空，索引 | 所属用户 |
| name | String(128) | 非空 | 患者姓名 |
| gender | String(16) | 可空 | 性别 |
| age | Integer | 可空 | 年龄 |
| phone | String(32) | 可空 | 联系方式 |
| height_cm | Integer | 可空 | 身高（cm） |
| weight_kg | Integer | 可空 | 体重（kg） |
| pain_regions | JSON | 可空 | 疼痛部位数组，如 `["腰部", "颈部"]` |
| history | Text | 可空 | 既往史 |
| allergy_history | Text | 可空 | 过敏史 |
| surgery_history | Text | 可空 | 手术史 |
| rehab_goal | Text | 可空 | 康复目标 |
| note | Text | 可空 | 备注 |
| created_at | DateTime | 默认当前时间 | 创建时间 |
| updated_at | DateTime | 自动更新 | 更新时间 |

**关联关系**：

- 被 `prescriptions`、`training_checkins`、`imaging_reports`、`wearable_metrics`、`doctor_patient_links`、`prescription_reviews` 引用

---

### 3.3 `prescriptions`（康复处方表）

**用途**：保存一次康复处方的生成结果，是系统**核心业务表**。

| 字段 | 类型 | 约束 / 索引 | 说明 |
|---|---|---|---|
| id | Integer | 主键，索引 | 处方 ID |
| user_id | Integer | 外键 `users.id`，可空，索引 | 创建处方的用户 |
| patient_profile_id | Integer | 外键 `patient_profiles.id`，可空，索引 | 关联患者档案 |
| patient_name | String(128) | 可空 | 患者姓名快照 |
| patient_age | Integer | 可空 | 患者年龄快照 |
| symptoms | Text | 非空 | 主诉 / 症状描述 |
| history | Text | 可空 | 病史 |
| summary | Text | 可空 | 大模型或本地兜底生成的处方摘要 |
| raw_response | JSON | 可空 | 模型原始响应、安全评估、RAG 上下文 |
| created_at | DateTime | 默认当前时间 | 创建时间 |
| updated_at | DateTime | 自动更新 | 更新时间 |

**关联关系**：

- `prescriptions` → `actions`（处方动作明细，1:N，级联删除）
- `prescriptions` → `training_checkins`（关联训练打卡，1:N）
- `prescriptions` → `prescription_reviews`（处方审核，1:N，级联删除）
- `prescriptions` → `prescription_adjustments`（处方调整，1:N）

---

### 3.4 `actions`（处方动作明细表）

**用途**：保存处方中实际推荐给患者执行的动作列表，是本次处方的执行快照。

| 字段 | 类型 | 约束 / 索引 | 说明 |
|---|---|---|---|
| id | Integer | 主键，索引 | 动作记录 ID |
| prescription_id | Integer | 外键 `prescriptions.id`，非空 | 所属处方 |
| name | String(128) | 非空 | 动作名称 |
| sets | Integer | 可空 | 组数 |
| reps | Integer | 可空 | 次数 |
| note | Text | 可空 | 动作说明或注意事项 |
| created_at | DateTime | 默认当前时间 | 创建时间 |
| updated_at | DateTime | 自动更新 | 更新时间 |

> 动作基础信息（禁忌、进阶、降阶、适用部位）来自 `knowledge/actions.json`，数据库只保存本次处方的执行快照。

---

### 3.5 `pose_feedback`（姿态反馈表）

**用途**：保存动作纠正接口的请求数据和反馈结果，便于追踪动作纠错记录。

| 字段 | 类型 | 约束 / 索引 | 说明 |
|---|---|---|---|
| id | Integer | 主键，索引 | 反馈 ID |
| request_data | JSON | 可空 | 动作 ID、关键点、可见度、时间戳等 |
| feedback | JSON | 非空 | 系统反馈文本、评分、状态 |
| created_at | DateTime | 默认当前时间 | 创建时间 |

---

### 3.6 `imaging_reports`（影像报告表）

**用途**：保存用户上传的影像报告、OCR 文本和系统识别出的红旗症状。

| 字段 | 类型 | 约束 / 索引 | 说明 |
|---|---|---|---|
| id | Integer | 主键，索引 | 报告 ID |
| user_id | Integer | 外键 `users.id`，非空，索引 | 所属用户 |
| patient_profile_id | Integer | 外键 `patient_profiles.id`，可空，索引 | 关联患者档案 |
| report_type | String(64) | 可空 | 报告类型，如 MRI、CT、X-Ray |
| file_name | String(256) | 可空 | 原始文件名 |
| file_path | String(512) | 可空 | 本地或 MinIO 文件路径 |
| ocr_text | Text | 可空 | OCR 或 TXT 提取文本 |
| ocr_status | String(32) | 非空，默认 `pending` | OCR 状态：`pending`/`done`/`failed` |
| risk_level | String(32) | 非空，默认 `unknown` | 风险等级：`low`/`medium`/`high`/`unknown` |
| red_flags | JSON | 可空 | 红旗症状识别结果列表 |
| note | Text | 可空 | 备注 |
| created_at | DateTime | 默认当前时间 | 创建时间 |
| updated_at | DateTime | 自动更新 | 更新时间 |

**业务逻辑**：处方生成前系统会读取患者影像风险等级，`high` 风险时阻止普通居家康复处方生成，强制要求医生介入。

---

### 3.7 `training_checkins`（训练打卡表）

**用途**：保存患者每次训练打卡记录，是趋势统计和康复进度报告的数据来源。

| 字段 | 类型 | 约束 / 索引 | 说明 |
|---|---|---|---|
| id | Integer | 主键，索引 | 打卡 ID |
| user_id | Integer | 外键 `users.id`，非空，索引 | 所属用户 |
| patient_profile_id | Integer | 外键 `patient_profiles.id`，可空，索引 | 关联患者档案 |
| prescription_id | Integer | 外键 `prescriptions.id`，可空，索引 | 关联处方 |
| action_id | String(128) | 可空 | 动作库 ID |
| action_name | String(128) | 非空 | 动作名称 |
| trained_on | Date | 非空，索引 | 训练日期 |
| completed_sets | Integer | 可空 | 完成组数 |
| completed_reps | Integer | 可空 | 完成次数 |
| pain_before | Integer | 可空 | 训练前 VAS 疼痛评分（0~10） |
| pain_after | Integer | 可空 | 训练后 VAS 疼痛评分（0~10） |
| difficulty | Integer | 可空 | 主观难度评分（0~10） |
| score | Integer | 可空 | 训练综合评分 |
| note | Text | 可空 | 备注 |
| created_at | DateTime | 默认当前时间 | 创建时间 |
| updated_at | DateTime | 自动更新 | 更新时间 |

**业务逻辑**：系统自动处方调整会读取近 14 天训练记录，结合疲劳数据判断是否需要降低训练负荷。

### 3.8 `wearable_metrics`（穿戴指标表）

**用途**：保存穿戴设备或用户手动录入的运动生理指标，系统计算疲劳风险并给出建议。

| 字段 | 类型 | 约束 / 索引 | 说明 |
|---|---|---|---|
| id | Integer | 主键，索引 | 指标 ID |
| user_id | Integer | 外键 `users.id`，非空，索引 | 所属用户 |
| patient_profile_id | Integer | 外键 `patient_profiles.id`，可空，索引 | 患者档案 |
| training_checkin_id | Integer | 外键 `training_checkins.id`，可空，索引 | 关联训练打卡 |
| device_type | String(64) | 可空 | 设备类型，如 Apple Watch、模拟录入 |
| heart_rate | Integer | 可空 | 运动心率（bpm） |
| resting_heart_rate | Integer | 可空 | 静息心率（bpm） |
| hrv_ms | Integer | 可空 | HRV（ms） |
| spo2 | Integer | 可空 | 血氧饱和度（%） |
| steps | Integer | 可空 | 步数 |
| calories | Integer | 可空 | 热量消耗（kcal） |
| skin_temperature_c | Float | 可空 | 皮温（℃） |
| perceived_exertion | Integer | 可空 | 主观用力程度（Borg 量表，0~20） |
| duration_minutes | Integer | 可空 | 训练时长（分钟） |
| fatigue_score | Integer | 非空，默认 0 | 系统综合疲劳评分 |
| risk_level | String(32) | 非空，默认 `low`，索引 | 疲劳风险等级：`low`/`medium`/`high` |
| signals | JSON | 可空 | 触发风险的具体指标信号 |
| recommendation | Text | 可空 | 系统给出的训练调整建议 |
| recorded_at | DateTime | 默认当前时间，索引 | 记录时间 |
| created_at | DateTime | 默认当前时间 | 创建时间 |

**疲劳算法**：综合心率、静息心率、HRV、血氧、主观用力程度和训练时长等多维指标，输出 `fatigue_score`、`risk_level`、`signals` 和 `recommendation`。

**用途**：保存医生或系统对处方提出的调整建议，以及患者的采纳 / 拒绝决策。

| 字段 | 类型 | 约束 / 索引 | 说明 |

## 6. 系统功能需求

**需求规划**：
- 用户可以注册账号，填写账号、密码和昵称。
- 系统应区分普通用户（患者）、医生和管理员三类角色。
- 密码必须哈希加盐保存，不能明文存储。
- 登录成功后发放鉴权 Token，后续接口通过 Token 验证身份。

**已实现情况**：

| 接口 | 方法 | 说明 |
|---|---|---|
| `/api/register` | POST | 注册账号，密码 PBKDF2-HMAC-SHA256 加盐哈希 |
| `/api/login` | POST | 登录，返回自定义 JWT Token |

- 角色通过 `users.role` 字段区分，支持 `user`、`doctor`、`admin`。
- 管理员和医生账号可由环境变量 `ADMIN_ACCOUNTS`、`DOCTOR_ACCOUNTS` 预配置。
- 后续所有需要鉴权的接口通过 `Authorization: Bearer <token>` 传递 Token。

- 检测到高风险红旗症状时，阻止生成普通居家康复处方，提示就医。

**已实现情况**：

| 接口 | 方法 | 说明 |
|---|---|---|

### 6.4 个性化康复处方生成

**需求规划**：
- 根据患者主诉、疼痛部位、活动度评分、病史和患者档案，生成个性化处方。
- 处方包含摘要、动作列表、组数、次数、注意事项。
- 结合安全规则和知识库，避免推荐高风险动作；高风险时阻止处方生成。

**已实现情况**：

| 接口 | 方法 | 说明 |
|---|---|---|
| `/api/generate_prescription` | POST | 生成个性化康复处方 |

处方生成链路：

```
用户提交主诉、疼痛部位、活动度评分
  → 读取患者档案与影像风险等级
  → validators 检测红旗症状（高风险则拦截）
  → RAG 检索康复知识上下文（knowledge/actions.json + 向量索引）
  → safety 模块过滤或降阶高风险动作
  → DeepSeek 大模型生成处方摘要（失败时本地兜底）
  → 保存 prescriptions 表和 actions 表
  → 返回处方摘要与动作列表
```

- 动作知识库：`knowledge/actions.json`，当前覆盖 16 个康复动作（颈部、肩部、腰部、膝关节、踝关节、髋部等）
- 大模型：DeepSeek / 火山方舟豆包（OpenAI 兼容接口）
- 兜底机制：大模型调用失败时，使用 `knowledge/prompt_template.txt` 本地规则生成可演示处方摘要

## 7. 核心数据流

### 7.1 注册登录数据流

```text
前端注册/登录表单
  → POST /api/register 或 /api/login
  → 校验账号、密码、昵称
  → 密码 PBKDF2-HMAC-SHA256 加盐哈希
  → 写入 users 表（含角色）
  → 登录成功返回 JWT Token
  → 后续请求：Authorization: Bearer <token>
```

### 7.2 患者档案与处方生成数据流

```text
患者创建档案（patient_profiles）
  → 提交主诉、疼痛部位、活动度评分
  → 读取患者档案 + 影像报告风险等级
  → validators 检测红旗症状 → 高风险则阻止
  → RAG 检索康复知识上下文
  → knowledge/actions.json 筛选候选动作
  → safety 模块过滤高风险动作
  → DeepSeek 生成处方摘要（失败走本地兜底）
  → 写入 prescriptions 表（含 raw_response）
  → 写入 actions 表（动作明细）
  → 前端展示处方和训练动作列表
```

### 7.3 影像报告风险数据流

```text
上传影像报告或录入 OCR 文本
  → object_storage 保存文件（本地/MinIO）
  → 写入 imaging_reports（file_path + ocr_text）
  → validators 检测红旗症状
  → 更新 red_flags 和 risk_level
  → 处方生成前读取风险等级
  → risk_level=high 时拦截处方生成，提示就医
```

### 7.4 训练打卡与报告数据流

```text
患者完成训练后打卡
  → 写入 training_checkins（动作、日期、完成量、VAS 疼痛评分）
  → /trends 接口按日期聚合打卡数据
  → /visualization 接口生成可视化统计
  → progress_reports 生成周报/月报
  → /report/export 导出 Markdown 报告
```

### 7.5 姿态纠正数据流

```text
前端摄像头/模拟关键点（33 个 MediaPipe 关键点）
  → POST /api/correct_pose 或 /api/pose/infer_frame
  → algorithms / pose_runtime 计算角度和反馈
  → 写入 pose_feedback 表
  → /api/voice/cue 生成语音提示文本
  → /api/visual/skeleton/frame 生成 3D 骨骼
  → /api/visual/ar/overlay 生成 AR 叠加信息
  → 前端展示反馈、评分和 AR 效果
```

### 7.6 医患协同与处方调整数据流

```text
患者绑定医生 → doctor_patient_links（status=active）
  → 患者分享处方 → prescription_reviews（status=pending）
  → 医生审核 → 更新 prescription_reviews（status=reviewed）
  → 医生提出调整 → prescription_adjustments（source=doctor, status=proposed）
  → 系统也可自动分析 → prescription_adjustments（source=system）
  → 患者采纳 → 生成新 prescriptions + actions
  → 患者拒绝 → status=rejected
```

### 7.7 生产部署数据流

```text
FastAPI 后端
  ├─ PostgreSQL 保存核心业务数据
  ├─ Redis 提供热点缓存和健康检查
  ├─ MinIO 保存影像报告等对象文件
  ├─ Chroma/Qdrant/local 保存 RAG 向量索引
  ├─ DeepSeek/火山方舟提供大模型处方生成
  └─ /ready + /api/deployment/info 检查服务状态
```

## 8. 非功能需求

### 8.1 安全性

- 密码必须哈希保存（PBKDF2-HMAC-SHA256 加盐），禁止明文存储。
- JWT Secret 通过环境变量配置，禁止提交到代码仓库。
- `.env` 保存私密配置，`.env.sample` 只保留示例占位符。
- 处方生成前进行红旗症状检测，降低医疗安全风险。
- 医生审核必须先建立 `active` 医患关系，防止越权访问患者处方。
- 管理接口（`/api/admin/*`）仅 `admin` 角色可访问。
- 影像报告文件下载需鉴权，确保只有本人或授权医生可访问（待完善）。

### 8.2 可维护性

- 数据模型统一在 SQLAlchemy ORM（`models.py`）中定义，便于集中管理。
- 请求和响应模型统一在 Pydantic schema（`schema.py`）中定义，保证接口文档准确。
- 数据库、缓存、对象存储、RAG、疲劳算法、进度报告、后台管理等模块按职责分层封装。
- 自动化测试脚本覆盖主要后端流程，生成 Markdown / JSON 报告便于验收。

### 8.3 可扩展性

- SQLite → PostgreSQL 只需修改 `DATABASE_URL`，无需改动业务代码。
- 本地文件存储 → MinIO 通过 `OBJECT_STORAGE_BACKEND` 切换。
- RAG 支持 local、Chroma、Qdrant 多后端，按需切换。
- 动作知识库使用 JSON 结构，便于管理员维护和后续扩充。
- 医患协同和处方调整表已为多轮审核、系统自动建议和新处方生成预留扩展结构。

### 8.4 可观测性

- `/health`：快速检查应用是否运行。
- `/ready`：检查数据库、Redis、对象存储就绪状态，供 Kubernetes 等平台使用。
- `/api/deployment/info`：返回部署环境和存储栈状态，不暴露敏感密钥。
- 后端自动化测试报告（`/api/admin/test_report`）可作为项目验收和演示证据。

## 10. 后续优化建议

1. **引入 Alembic 数据库迁移**：替代手写 `_ensure_sqlite_columns()` 补列逻辑，支持生产环境安全升级。
2. **细化 PostgreSQL 复合索引**：如 `(user_id, created_at)`、`(patient_profile_id, trained_on)`，提升大数据量下的查询性能。
3. **处方与审核状态枚举约束**：为 `prescriptions`、`prescription_reviews`、`prescription_adjustments` 的 `status` 字段增加 CHECK 约束或枚举类型。
4. **影像报告下载鉴权**：确保报告只能被本人或授权医生通过 Token 下载，防止直链泄露。
5. **RAG 知识条目管理表**：新增 `knowledge_articles` 表，使知识库不仅依赖 JSON 文件，也可通过后台界面维护和扩充。
6. **审计日志表**：新增 `audit_logs` 表，记录管理员操作、医生审核和敏感数据访问，满足合规要求。
7. **数据备份与恢复策略**：生产环境定期备份 PostgreSQL、MinIO bucket 和向量索引，制定 RTO/RPO 目标。
8. **隐私合规**：对患者健康数据、影像报告和训练记录增加访问控制说明和数据最小化展示策略。
9. **性能优化**：对 RAG 索引建立热点缓存（Redis），减少处方生成时的向量检索延迟。
10. **前端完善**：接入 MediaPipe Pose 实时关键点提取，替代当前模拟关键点演示模式。
